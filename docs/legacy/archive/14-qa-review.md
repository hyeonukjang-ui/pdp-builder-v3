# PDP 자동 생성 도구 — QA / Devil's Advocate 리뷰

> QA 리뷰어 관점 | 2026-03-16

---

## 1. 기술적 리스크

### 1-1. MRT 스크래핑 실동작 가능성 🔴

- CTO 문서에서 `__NEXT_DATA__` 파싱을 첫 전략으로 제시했으나, MRT 실조사 결과 **Next.js가 아니라 Rails SSR**임이 확인됨
- 실제 추출 가능한 소스는 `script[data-component-name="Offer"]` JSON
- 같은 회사라면 내부 API를 쓰는 게 맞지만, 프로토타입에서는 스크래핑이 현실적

**해결**: 실제 MRT URL 5개 이상으로 검증 필수. MRT 실조사 문서(10번)의 추출 전략으로 수정

### 1-2. ES Modules 브라우저 호환성 🟢

- 2026년 기준 97%+ 커버리지. 내부 도구이므로 문제없음
- 단, `package.json`에 `"type": "module"` 필요

### 1-3. Claude API 호출 시간 🟡

- CTO: "병렬 시 ~4-5초" vs UX: "전체 ~15초" — **두 문서 간 수치 3배 차이**
- 실제: 스크래핑 2-3초 + 분류 1-2초 + 카피 생성 5-10초 = **8-15초**
- progressive rendering은 문서에 언급되었지만 **실제 구현 방법 미정의** (`renderPDP()`는 한번에 전체 렌더)

**해결**:
1. 15초 대기 인정하고 스켈레톤 UI + 단계별 진행률에 집중
2. **데모는 프리셋으로 시작**, 라이브 URL은 보너스로

### 1-4. 에러 핸들링 누락 시나리오 🟡

빠진 것들:
- Claude API rate limit (429) 시 backoff 전략
- 전체 타임아웃 컷오프 미정의
- MRT 서버 다운/점검 시 처리
- Claude JSON 파싱 실패
- 이미지 URL CDN 외부 접근 차단

## 2. 아키텍처 갭

### 2-1. 코드 중복 🔴

렌더러가 **3벌** 존재:
1. `/script.js` — 프레젠테이션용 인라인 렌더 12개
2. `/presentation/script.js` — 100% 동일 복사본
3. `/blocks/*.js` — 모듈형 렌더러 12개

→ 현재 `_archive`로 이동됨. 재구현 시 **`blocks/*.js` 모듈형을 Single Source of Truth**로 확정

### 2-2. 레시피 18개 vs 구현 12개 블록 🔴

| 미구현 블록 | 사용 카테고리 수 |
|------------|:---:|
| reviews | 8개 전체 |
| socialProof | 8개 전체 |
| relatedProducts | 8개 전체 |
| notice (모듈형) | 8개 전체 |
| faq (모듈형) | 8개 전체 |
| cta (모듈형) | 8개 전체 |
| meetingPoint | 6개 |
| hotelInfo | 1개 (세미패키지) |

`pdp-engine.js`의 graceful skip으로 렌더러 없으면 건너뛰지만, **reviews가 빠지면 프로젝트 핵심 주장과 모순** (리뷰/UGC 전환율 영향 1위 +354%)

## 3. UX 문제점

### 3-1. 15초 대기 🟡

프로그레스 바의 66%~100% 구간(AI 카피 생성)이 5-10초라 멈춰 보임.

**해결**: 추출된 기본 정보(제목, 이미지, 가격)를 먼저 보여주면 "맞는 상품이구나" 확인하며 기다릴 수 있음

### 3-2. 샘플 데이터 유사도 🟡

- 8개 필요한데 현재 3개만 존재
- 기존 샘플은 Unsplash 이미지 + 이상적 데이터. 실제 MRT 상품은 불완전할 가능성 높음
- **"이상적 샘플" 외에 "현실적 샘플"도 하나 추가 권장** — graceful degradation 데모용

### 3-3. 프리뷰 vs 실제 MRT PDP 혼란 🟡

**해결**: 프리뷰 상단에 "설계 프리뷰 — 실제 배포 화면이 아닙니다" 워터마크

## 4. 카피 품질 리스크

### 4-1. AI 한국어 카피 자연스러움 🟡

- 글자 수 제한(title ≤ 25자, highlight ≤ 40자)을 AI가 정확히 지키기 어려움
- "최고의", "완벽한" 같은 검증 불가 최상급 표현 빠져나올 수 있음

**해결**: 생성 후 자동 검증 함수 (글자 수, 금칙어 정규식, 형식 체크)

### 4-2. 환각(Hallucination) 🔴 **가장 위험한 리스크**

AI가 상품에 없는 정보를 만들어낼 수 있음:
- "무료 취소" — 실제 환불 불가
- "호텔 픽업 포함" — 실제 미포함
- "한국어 가이드" — 실제 영어만

**해결 (필수)**:
1. 가격/취소정책/포함불포함은 **AI 생성 금지, 원본 데이터 그대로 사용**
2. socialProof도 실시간 데이터만 사용, AI 생성 금지
3. 생성 카피에 "팩트 체크 필요" 마커 자동 부착
4. 원본 데이터 vs AI 카피 병렬 비교 UI

### 4-3. 카테고리 오분류 🟡

경계 모호한 케이스 많음 (한강 유람선 = TICKET_EXPERIENCE? ACTIVITY?)

**해결**: confidence score 표시, 70% 미만이면 수동 선택 요청

## 5. 운영 리스크

### 5-1. MRT 프론트엔드 리뉴얼 시 스크래퍼 파손 🔴

스크래핑의 본질적 한계. 전면 리뉴얼 시 모든 전략이 동시에 깨질 수 있음.

**해결**: 단기 — health check 모니터링 / 중기 — 내부 API 전환

### 5-2. API 키 비용 🟢

데모 ~$3.5/월, 풀 스케일 ~$175/월. 현실적이고 합리적.

### 5-3. 데모 중 장애 🟡

**해결**: 데모 순서를 **프리셋 → 카테고리 전환 → 라이브 URL(보너스)** 역순으로

## 6. 누락된 기능

### 6-1. 생성 결과 저장/공유 🟡

MVP: localStorage 자동 저장. P1: JSON 다운로드/업로드. P2: 공유 URL.

### 6-2. 카피 수정 기능 🔴

"80% 자동 + 20% 검수" 운영 모델인데 검수 후 수정할 방법이 없음.

**해결**: MVP에서 최소한 "JSON 편집 모드" 토글 필요

### 6-3. 이미지 프록시 🟡

MRT CDN이 외부 핫링크를 차단할 수 있음.

**해결**: 백엔드에 `/api/image-proxy?url=...` 엔드포인트

### 6-4. 접근성 🟡

생성되는 PDP HTML의 a11y가 더 중요 (블록 렌더러에 ARIA 추가)

## 7. 전체 판정

### 강점 3가지

1. **전략적 프레이밍**: "콘텐츠를 채우는 게 아니라 전환을 설계한다" — 설득력 높음
2. **렌더링 엔진 아키텍처**: block-registry + recipes + engine 3파일 구조가 깔끔. 새 블록 추가가 파일 1개로 가능
3. **비용 효율**: Haiku+Sonnet 분리, PDP당 $0.02~$0.07. 기존 수동 제작(2-5일) 대비 압도적 ROI

### 약점 3가지

1. **스크래핑 미검증**: 전체 파이프라인 첫 단계가 이론적 설계에 그침 → MRT 실조사로 해결됨
2. **품질 검증/수정 레이어 부재**: 환각 체크, 카피 가이드라인 자동 검증, 수동 수정 기능 모두 없음
3. **설계와 구현 간 갭**: 18개 블록 계획인데 12개 구현, 8개 샘플 필요한데 3개
