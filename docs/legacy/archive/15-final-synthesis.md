## 전문가 간 충돌 해소

### 충돌 1: MRT 스크래핑 전략

- **CTO 제안**: `__NEXT_DATA__` → window 전역변수 → OG 태그 → Puppeteer
- **MRT 리서처 실조사 결과**: MRT는 **Next.js가 아니라 Rails SSR + React on Rails**. `__NEXT_DATA__` 존재하지 않음.

**→ 최종 결정**: `script[data-component-name="Offer"]` JSON 파싱이 메인 전략. Puppeteer 불필요.

```
확정된 추출 전략:
1순위: data-component-name="Offer" JSON 파싱 (Rails SSR)
2순위: OG 메타 태그 (부분 데이터)
3순위: HTML 구조 파싱 (최후 수단)
```

### 충돌 2: AI 호출 시간

- **CTO**: "병렬 시 ~4-5초"
- **UX 디자이너**: "전체 ~15초"

**→ 최종 결정**: 실제는 **8-15초** (스크래핑 2-3초 + 분류 1초 + 카피 생성 5-10초). UX 문서 기준이 현실적. 데모에서는 프리셋으로 시작하고 라이브 URL은 보너스로 위치.

### 충돌 3: AI 호출 패턴

- **CTO**: 블록 그룹별 5회 분할 (분류 1 + 생성 4)
- **카피라이터**: 2단계 하이브리드 (브리프 Sonnet 1 + 블록별 Haiku N)

**→ 최종 결정**: 카피라이터 안 채택. 이유:
1. 브리프 단계가 블록 간 톤 일관성을 보장
2. 블록별 실패가 격리됨
3. Haiku가 더 빠르고 저렴 (전체 비용 ~$0.02~0.04)

### 충돌 4: 환각 처리

- **CTO**: 에러 핸들링만 (블록 건너뛰기)
- **QA 리뷰어**: 가장 위험한 리스크로 지적
- **카피라이터**: AI 생성 영역 3단계 분류 제안

**→ 최종 결정**: 3단계 분류 채택 (필수)
- **AI 생성 금지**: 가격, 취소정책, 포함/불포함 → 원본 데이터 직접 사용
- **AI 다듬기만**: inclusions 톤, optionTable 형식
- **AI 자유 생성**: hero 부제, highlights, overview, recommendFor

## 확정된 기술 스택

| 영역 | 기술 | 이유 |
|------|------|------|
| 백엔드 | Node.js + Express | 프론트와 동일 언어, 빠른 프로토타이핑 |
| 스크래핑 | Cheerio | SSR HTML 파싱 충분, Puppeteer 불필요 |
| AI 분류 | Claude Haiku | 빠르고 저렴, 단순 분류 |
| AI 카피 | Claude Sonnet (브리프) + Haiku (블록별) | 품질+속도+비용 균형 |
| 프론트엔드 | 바닐라 JS + ES Modules | 빌드 불필요, 기존 코드 재활용 |
| 스타일 | Pretendard, MRT 컬러 토큰 | 브랜드 일관성 |

## P0~P2 이슈 (QA 리뷰 기반 확정)

### P0 — MVP 필수

1. **MRT URL 5개+ 실제 스크래핑 검증** — 라이브 데모 성패 좌우
2. **환각 방지 — 가격/정책 AI 생성 금지** — 고객 피해 방지
3. **8개 카테고리 샘플 데이터** — "8개 지원"인데 3개만 보여줄 수 없음
4. **reviews/notice/faq/cta 블록 구현** — 핵심 블록 누락 불가

### P1 — 다음 스프린트

1. 카피 수정(인라인 편집) 기능
2. meetingPoint, socialProof, relatedProducts 블록
3. 카피 품질 자동 검증 함수
4. 결과 localStorage 저장

### P2 — 나중에 해도 됨

1. Progressive rendering (SSE)
2. 이미지 프록시
3. 접근성
4. 공유 URL
5. MRT 내부 API 연동

## 구현 순서 (확정)

```
Phase 1: 기반 구축
├── package.json + Express 서버 뼈대
├── 정적 파일 서빙 (ES Modules)
├── .env + Anthropic SDK 셋업
└── 디렉토리 구조 생성

Phase 2: 데이터 파이프라인
├── lib/extractor.js — MRT Cheerio 스크래핑
├── lib/field-mapper.js — 원본 → 블록 스키마 변환
├── lib/generator.js — Claude 카피 생성 (브리프 + 블록)
├── POST /api/extract, /api/generate 라우트
└── 실제 MRT URL 5개 검증 (P0-1)

Phase 3: 렌더링 엔진 + 블록
├── engine/ 재구축 (기존 코드 기반, 정리)
├── blocks/ 12개 기존 블록 (스키마 통일)
├── blocks/ 8개 신규 블록 (reviews, notice, faq, cta, ...)
├── recipes/ 8개 카테고리 레시피
└── 8개 샘플 데이터 JSON (P0-3)

Phase 4: 도구 UI
├── index.html — 사이드바 + 프리뷰 레이아웃
├── style.css — 디자인 토큰 + 컴포넌트
├── app.js — URL 입력 → API → 렌더링 연결
├── 상태별 UI (empty, loading, success, error, offline)
└── 통합 테스트 + 데모 시나리오
```

## 데모 시나리오 (확정)

```
1단계: 프리셋 데모 (확실하게 동작)
→ 8개 카테고리 샘플 즉시 전환
→ "각 카테고리별로 블록 구성이 다릅니다"

2단계: 카테고리 전환 데모 (확실하게 동작)
→ 같은 데이터, 다른 레시피로 블록 순서 변경
→ "같은 상품도 카테고리에 따라 정보 배치가 달라집니다"

3단계: 라이브 URL 데모 (보너스)
→ 실제 MRT URL 입력 → 실시간 생성
→ 사전 테스트 완료된 URL 사용
→ 실패 시 자연스럽게 프리셋으로 전환
```
